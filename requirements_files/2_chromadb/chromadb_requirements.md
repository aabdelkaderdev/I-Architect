## 1. Functional Requirements (FR)

### FR-CHROMA-001: Build-Time Knowledge Ingestion Pipeline
- **Description:** The system SHALL execute a Python seeding script (`seed_knowledge_base.py`) during the Docker image build process that ingests source PDF documents from the `/knowledge_source/` build context directory, processes them through text extraction, chunking, and embedding generation, and persists the resulting vector index into the `/chroma_data` directory within the Docker image layer.
- **SRS Trace:** §10.4, §10.1
- **Priority:** Must
- **Acceptance Criteria:** After `docker build`, the resulting image contains a fully populated `/chroma_data` directory with all four standardized collections queryable without runtime internet access or additional seeding steps.

### FR-CHROMA-002: PDF Text Extraction
- **Description:** The seeding script SHALL extract text from source PDF documents using `pypdf` or `pdfplumber`, discarding non-textual content (images, charts, embedded media).
- **SRS Trace:** §10.4
- **Priority:** Must
- **Acceptance Criteria:** All textual content from each source PDF is extracted into a raw text stream with page-number metadata preserved.

### FR-CHROMA-003: Recursive Character Text Splitting
- **Description:** The seeding script SHALL split extracted text using `RecursiveCharacterTextSplitter` with a target chunk size of 512 characters and an overlap of 50 characters to preserve contextual continuity across chunk boundaries.
- **SRS Trace:** §10.4
- **Priority:** Must
- **Acceptance Criteria:** Each chunk is ≤ 512 characters with a 50-character overlap with its adjacent chunk; no text is lost during splitting.

### FR-CHROMA-004: Embedding Generation with Standardized Model
- **Description:** The seeding script SHALL compute vector embeddings using the `sentence-transformers/all-MiniLM-L6-v2` model, producing 384-dimensional vectors for each text chunk.
- **SRS Trace:** §10.4
- **Priority:** Must
- **Acceptance Criteria:** All stored vectors have exactly 384 dimensions and are generated by the `all-MiniLM-L6-v2` model.

### FR-CHROMA-005: Four Standardized Collections Schema
- **Description:** The database SHALL be pre-populated with exactly four global collections, each using Cosine Similarity as the distance metric:
  1. `c4_model_standards` — Rules for C4 Context, Container, and Component diagrams.
  2. `eip_patterns` — Enterprise Integration Pattern definitions (Messaging, Routing, Transformation).
  3. `plantuml_syntax` — Valid syntax examples for both C4-PlantUML stdlib and standard PlantUML.
  4. `saam_atam_criteria` — SAAM/ATAM evaluation scoring rubrics.
- **SRS Trace:** §10.4
- **Priority:** Must
- **Acceptance Criteria:** Querying each collection name via the ChromaDB API returns a valid, non-empty collection with documents and embeddings.

### FR-CHROMA-006: Collection Metadata Schema
- **Description:** Each document chunk stored in any collection SHALL include the following metadata fields:
  - `source_document` (string): The originating PDF filename.
  - `page_number` (integer): The page from which the chunk was extracted.
  - `category` (string): One of `"Syntax"`, `"Pattern"`, or `"Rule"`.
- **SRS Trace:** §10.4
- **Priority:** Must
- **Acceptance Criteria:** Every stored document in every collection contains all three metadata fields with valid, non-null values.

### FR-CHROMA-007: HTTP API Exposure
- **Description:** The container SHALL run the ChromaDB server in HTTP mode, binding to `0.0.0.0` on internal port `8000`, exposed to the host on port `8200`. The API SHALL be accessible to all containers on the `i-architect-net` Docker bridge network without authentication for read operations.
- **SRS Trace:** §10.1, §10.4
- **Priority:** Must
- **Acceptance Criteria:** `GET http://localhost:8200/api/v1/heartbeat` returns HTTP `200 OK`; agents within the Docker network can query collections via `http://i-architect-chromadb:8000`.

### FR-CHROMA-008: Health Check Endpoint
- **Description:** The container SHALL expose a health check endpoint at `GET /api/v1/heartbeat` that returns HTTP `200 OK` when the server is ready and the vector index is loaded.
- **SRS Trace:** §10.1
- **Priority:** Must
- **Acceptance Criteria:** The orchestrator's startup health check receives a `200 OK` response before marking the pipeline as "Ready."

### FR-CHROMA-009: Source Document Ingestion Set
- **Description:** The build pipeline SHALL ingest exactly the following four PDF source documents from the `/knowledge_source/` directory:
  1. `C4_Model_Specification.pdf`
  2. `Enterprise_Integration_Patterns_Ref.pdf`
  3. `PlantUML_Language_Reference_Guide.pdf`
  4. `Architecture_Tradeoff_Analysis_Method.pdf`
- **SRS Trace:** §10.4
- **Priority:** Must
- **Acceptance Criteria:** The build fails if any of the four required PDFs is missing from the build context.

### FR-CHROMA-010: Agent Collection Access Mapping
- **Description:** The system SHALL enforce the following agent-to-collection access mapping:
  - **RAA:** `c4_model_standards`, `eip_patterns`
  - **AGA:** `plantuml_syntax`
  - **SA:** `c4_model_standards`, `saam_atam_criteria`, `plantuml_syntax`
- **SRS Trace:** §10.4
- **Priority:** Should
- **Acceptance Criteria:** Each agent queries only its designated collections; query logs confirm no cross-collection violations.

## 2. Non-Functional Requirements (NFR)

### NFR-CHROMA-001: Query Latency
- **Description:** The system SHALL return search results (k=5 nearest neighbors) within 500ms for 95% of queries under normal operating conditions.
- **SRS Trace:** §10.4
- **Metric:** P95 query latency ≤ 500ms (including query embedding computation and HNSW index search).

### NFR-CHROMA-002: Memory Footprint
- **Description:** The running container (ChromaDB server + loaded HNSW index + embedding model) SHALL NOT exceed 1GB RAM as enforced by the Docker Compose resource limit.
- **SRS Trace:** §10.1
- **Metric:** `docker stats` reports peak memory usage ≤ 1024MB under sustained concurrent query load.

### NFR-CHROMA-003: Immutability and Factory Reset
- **Description:** The `/chroma_data` directory SHALL be part of the immutable Docker image layer (not a Docker Volume). Restarting the container SHALL restore the knowledge base to its exact build-time state, resolving any potential runtime corruption or accidental writes.
- **SRS Trace:** §10.1
- **Metric:** After container restart, the SHA-256 checksums of all collection data files match the original build-time checksums.

### NFR-CHROMA-004: Multi-Stage Docker Build Optimization
- **Description:** The Docker image SHALL use a multi-stage build to minimize final image size by discarding intermediate build tools (compilers, pip caches, PyTorch build artifacts) while retaining the model weights, vector index, and ChromaDB server runtime.
- **SRS Trace:** §10.1
- **Metric:** Final image size ≤ 2.5GB.

### NFR-CHROMA-005: Concurrent Query Support
- **Description:** The ChromaDB server SHALL handle concurrent queries from up to 3 parallel agent instances (Workflows 2 & 3) without query failures or exceeding the 500ms latency SLA.
- **SRS Trace:** §10.4, §9.1
- **Metric:** Under 3 simultaneous k=5 queries, P95 latency remains ≤ 500ms with zero error responses.

### NFR-CHROMA-006: Zero-Configuration Deployment
- **Description:** The knowledge base SHALL require no user-initiated seeding, downloading, or configuration on first launch. The command `docker compose up` SHALL be the only step required to have a fully operational ChromaDB instance.
- **SRS Trace:** §10.1
- **Metric:** A fresh `docker compose up` from a clean pull results in a healthy ChromaDB instance within 60 seconds.

## 3. Interface Requirements (IR)

### IR-CHROMA-001: Internal Agent Query Interface
- **Endpoint:** `http://i-architect-chromadb:8000`
- **Protocol:** HTTP (ChromaDB Python Client SDK)
- **Read Operation:** `collection.query(query_texts=["..."], n_results=5)`
- **Write Operation:** Not supported at runtime (changes are ephemeral and lost on restart).
- **Authentication:** None required for read operations within the `i-architect-net` network.

### IR-CHROMA-002: External Debug Interface
- **Endpoint:** `http://localhost:8200/api/v1`
- **Protocol:** HTTP REST
- **Purpose:** Developer verification of knowledge base state and collection contents.
- **Health Check:** `GET /api/v1/heartbeat` → `200 OK`.

### IR-CHROMA-003: Docker Network Configuration
- **Network:** `i-architect-net` (bridge)
- **Service Name:** `i-architect-chromadb`
- **Internal Port:** `8000`
- **Host Port:** `8200`

## 4. Disaster Recovery Requirements (DR)

### DR-CHROMA-001: Container Restart Recovery
- **Failure Mode:** ChromaDB process crash, memory corruption, or unexpected container stop.
- **Recovery Action:** Docker's `restart: on-failure` policy automatically restarts the container. Since data is baked into the image layer, a restart is effectively a factory reset — no data loss is possible.
- **SRS Trace:** §13

### DR-CHROMA-002: RAG Retrieval Failure Fallback
- **Failure Mode:** ChromaDB is unreachable (`ConnectionError`) or returns zero results for a critical agent query.
- **Recovery Action:** The requesting agent (RAA, AGA, or SA) SHALL fall back to embedded few-shot examples hardcoded in the prompt template. The system SHALL log the RAG failure event.
- **User-Facing Message:** ⚠️ *"The architectural knowledge base is temporarily unavailable. Analysis will proceed using built-in templates."*
- **SRS Trace:** §13.3 (R-2)

### DR-CHROMA-003: Image Rebuild for Knowledge Updates
- **Failure Mode:** Knowledge base contains stale or outdated architectural standards.
- **Recovery Action:** A new CI/CD pipeline run rebuilds the Docker image with updated source PDFs. No runtime update mechanism exists.
- **SRS Trace:** §10.4
