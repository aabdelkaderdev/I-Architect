## 1. Functional Requirements (FR)

### 1.1 Data Ingestion & Parsing

#### FR-ARLO-001: Static Quality-Architecture Pattern Matrix Loading
- **Description:** The system SHALL load the `quality_archipattern_matrix.csv` static asset at startup. The matrix maps architectural patterns (rows) to quality attributes (columns) with integer impact scores. Rows are grouped by architectural decision categories. The matrix SHALL support both tab-separated and comma-separated formats.
- **SRS Trace:** §9.3
- **Legacy Trace:** FR-TDM-006, FR-TDM-007, FR-TDM-008, FR-TDM-009
- **Priority:** Must
- **Acceptance Criteria:** The matrix loads with correct row grouping (7 categories, mutually exclusive patterns per category) and QA acronym resolution (PE, CO, US, RE, SE, MA, PO, CE).

#### FR-ARLO-002: Requirements Ingestion from Structured CSV
- **Description:** ARLO SHALL ingest the pre-processed requirements from `/{project}/structured_requirements/requirements.csv` (output of the Filtering Agent/Editor). The system SHALL inherit Requirement IDs from the upstream pipeline — it SHALL NOT auto-assign its own sequential integer identifiers.
- **SRS Trace:** §9.3
- **Legacy Trace:** FR-RP-001, FR-RP-002 (migration: override auto-ID with pipeline-provided ID)
- **Priority:** Must
- **Acceptance Criteria:** ARLO processes requirements with IDs like `REQ-a1b2c3-001` (pipeline format), not legacy `1, 2, 3` (integer format).

#### FR-ARLO-003: User Override Enforcement
- **Description:** ARLO SHALL check the `user_override` column before processing any requirement. If `user_override == True`, the requirement SHALL ALWAYS be preserved regardless of `is_noisy` value. ARLO SHALL NEVER re-drop user-overridden rows during its ingestion phase.
- **SRS Trace:** §9.3
- **Legacy Trace:** N/A (new feature in v2.3)
- **Priority:** Must
- **Acceptance Criteria:** A requirement with `user_override=True` and `is_noisy=True` is included in ARLO's ASR analysis and never filtered out.

#### FR-ARLO-004: Noise Dropping Mechanism
- **Description:** ARLO SHALL scan the `is_noisy` column. If `is_noisy=True` AND `user_override=False`, ARLO SHALL drop the row from memory entirely before the ASR Identification step. This conserves API tokens by reducing the input set.
- **SRS Trace:** §9.3
- **Legacy Trace:** N/A (new pre-filter integration in v2.3)
- **Priority:** Must
- **Acceptance Criteria:** A set of 150 requirements with 30 noisy (no overrides) rows results in ARLO processing exactly 120 requirements.

#### FR-ARLO-005: CSV to ARLO Entity Schema Mapping
- **Description:** ARLO SHALL map the incoming CSV schema to its internal model:
  | CSV Column | ARLO Internal | Mapping Rule |
  |:--|:--|:--|
  | `Requirement ID` | Unique identifier | Direct Map (overrides legacy auto-integer) |
  | `Raw Description` | Description | Direct Map |
  | `Source Context` | *New Extension* | Appended for traceability |
  | `user_override` | Override Flag | Direct Map |
  | *(Generated by LLM)* | IsArchitecturallySignificant | Initialized as Null |
  | *(Generated by LLM)* | Quality Attributes | Initialized as empty list |
  | *(Generated by LLM)* | Condition Text | Initialized as "under any circumstances" |
- **SRS Trace:** §9.3
- **Legacy Trace:** FR-MOD-001 (migration: extended model with `user_override` and `Source Context`)
- **Priority:** Must
- **Acceptance Criteria:** The ARLO entity model contains all fields from the table above, with legacy integer ID replaced by pipeline string ID.

### 1.2 ASR Identification & QA Classification

#### FR-ARLO-006: LLM-Based ASR Identification (Stringent Mode Only)
- **Description:** ARLO SHALL use the LLM to determine whether each requirement is Architecturally Significant. V2.3 SHALL use ONLY the "Stringent" classification mode (the legacy "Lax" toggle is removed). A requirement is ASR IF AND ONLY IF it: (a) explicitly dictates a high-level design choice, AND (b) specifies one or more of the 8 recognized QAs.
- **SRS Trace:** §9.3
- **Legacy Trace:** FR-RP-003, FR-RP-004 (migration: Lax mode removed; always Stringent)
- **Priority:** Must
- **Acceptance Criteria:** "The system shall use a Microservices architecture for independent scaling" is classified as ASR with QA `[Performance Efficiency, Maintainability]`.

#### FR-ARLO-007: Quality Attribute Extraction
- **Description:** For each requirement, ARLO SHALL extract a list of relevant quality attributes from the fixed taxonomy: Performance Efficiency (PE), Compatibility (CO), Usability (US), Reliability (RE), Security (SE), Maintainability (MA), Portability (PO), Cost Efficiency (CE). Only QAs from this predefined list SHALL be accepted.
- **SRS Trace:** §9.3
- **Legacy Trace:** FR-RP-006
- **Priority:** Must
- **Acceptance Criteria:** The LLM output is validated against the 8-QA enum; invalid QA values trigger retry per DR-ARLO-002.

#### FR-ARLO-008: Condition Text Extraction
- **Description:** For each requirement, ARLO SHALL extract conditional statements (e.g., "if bandwidth is low", "when traffic is high"). If no condition is present, the system SHALL default to "under any circumstances."
- **SRS Trace:** §9.3
- **Legacy Trace:** FR-RP-007
- **Priority:** Must
- **Acceptance Criteria:** "The system must cache data when internet is slow" extracts condition "when internet is slow."

#### FR-ARLO-009: Batch Processing with Token Management
- **Description:** ARLO SHALL send requirements to the LLM in batches of 10 (configurable). Before each batch, the system SHALL use `tiktoken` to calculate the token count and ensure it stays within context limits (replacing the legacy hardcoded 5000-word limit).
- **SRS Trace:** §9.3
- **Legacy Trace:** FR-RP-008, FR-RP-009, FR-LLM-006 (migration: tiktoken replaces word counting)
- **Priority:** Must
- **Acceptance Criteria:** A batch of 10 requirements with combined token count exceeding the model's limit is automatically split into sub-batches.

#### FR-ARLO-010: User Override QA Handling
- **Description:** If a requirement is forced by `user_override=True` but the LLM finds NO applicable QA (e.g., purely aesthetic constraint), the system SHALL mark it as `Type: CONSTRAINT` — excluded from the Objective Function (weight = 0) but included in the final report as a mandatory textual constraint.
- **SRS Trace:** §9.3
- **Priority:** Must
- **Acceptance Criteria:** A user-overridden requirement "Must use Blue color scheme" is marked CONSTRAINT, excluded from ILP optimization, but listed in the ARLO report.

### 1.3 Semantic Condition Grouping

#### FR-ARLO-011: Vector-Based Condition Grouping
- **Description:** ARLO SHALL replace the legacy TF-IDF + K-Means condition grouping with a semantic approach:
  1. Extract `ConditionText` from each requirement.
  2. Generate vector embeddings using `sentence-transformers/all-MiniLM-L6-v2` via the ChromaDB client.
  3. Perform Agglomerative Clustering on vectors (cosine similarity threshold > 0.85).
  4. Output distinct `SatisfiableGroups` — requirements in the same group must be satisfied simultaneously.
- **SRS Trace:** §9.3
- **Legacy Trace:** FR-CL-001–005, FR-EMB-001–005 (migration: TF-IDF/K-Means → sentence-transformers/ChromaDB)
- **Priority:** Must
- **Acceptance Criteria:** "When internet is slow" and "In low bandwidth scenarios" are grouped together; "During peak traffic" is in a separate group.

#### FR-ARLO-012: Condition-less Requirements Grouping
- **Description:** Requirements without a specific condition (condition = "under any circumstances") SHALL be placed into a single dedicated Condition Group that is included in ALL Satisfiable Groups.
- **SRS Trace:** §9.3
- **Legacy Trace:** FR-AD-002, FR-AD-006(c)
- **Priority:** Must
- **Acceptance Criteria:** Unconditional requirements appear in every generated Satisfiable Group.

#### FR-ARLO-013: LangChain Caching via SQLiteCache
- **Description:** ARLO's custom file-based JSON caching SHALL be replaced by LangChain's native `SQLiteCache` for parsed requirements and condition groups.
- **SRS Trace:** §9.3
- **Legacy Trace:** FR-RP-011, FR-AD-004 (migration: file-based cache → SQLiteCache)
- **Priority:** Must
- **Acceptance Criteria:** Repeated ARLO runs with identical input hit the SQLite cache instead of re-invoking the LLM.

### 1.4 Mathematical Optimization

#### FR-ARLO-014: Quality Weight Inference
- **Description:** ARLO SHALL infer QA weights based on requirement frequency: `W_qa = Count(Requirements_qa) / Total(ArchitecturalRequirements)`. If the total count is too low (< 3), the system SHALL default to equal weights. User-provided weights via configuration SHALL also be supported.
- **SRS Trace:** §9.3
- **Legacy Trace:** FR-AD-010, FR-AD-011
- **Priority:** Must
- **Acceptance Criteria:** Given 10 ASRs where 5 mention PE and 3 mention SE, `W_PE = 0.5` and `W_SE = 0.3` (normalized).

#### FR-ARLO-015: ILP Optimization Engine
- **Description:** ARLO SHALL construct an Integer Linear Programming model using Google OR-Tools (Python) or PuLP:
  - **Variables:** `x_p` (binary; 1 if pattern `p` is selected).
  - **Objective:** Maximize `Σ(x_p × Score(p,qa) × W_qa)`.
  - **Constraints:** Exactly one pattern per architectural decision category (mutually exclusive).
  - Support up to 100 simultaneous variables (patterns) and 500 constraints without exceeding a 30-second timeout.
- **SRS Trace:** §9.3
- **Legacy Trace:** FR-OPT-001, FR-OPT-002, FR-OPT-003
- **Priority:** Must
- **Acceptance Criteria:** Given a 7-category matrix with 2–3 patterns each, ILP selects exactly one pattern per category and reports the weighted score.

#### FR-ARLO-016: Greedy Optimization Fallback
- **Description:** ARLO SHALL support a Greedy optimization algorithm as an alternative to ILP. The greedy algorithm independently selects the pattern with the highest weighted score from each decision category.
- **SRS Trace:** §9.3
- **Legacy Trace:** FR-OPT-004, FR-DM-001
- **Priority:** Must
- **Acceptance Criteria:** Greedy optimization produces a valid decision set within 1 second for any input size.

#### FR-ARLO-017: Optimization Strategy Selection (UI-Driven)
- **Description:** The user SHALL select between ILP and Greedy optimization strategies via the ARLO Page (Page 5) dropdown. The selection is passed as configuration to the ARLO pipeline.
- **SRS Trace:** §1.7, §9.3
- **Legacy Trace:** FR-OPT-005, FR-CFG-002
- **Priority:** Must
- **Acceptance Criteria:** Selecting "Greedy" in the UI results in ARLO using the Greedy solver instead of ILP.

### 1.5 Sensitivity Analysis (Beta Experiment)

#### FR-ARLO-018: Architecturally Influential Set Identification
- **Description:** ARLO SHALL execute sensitivity analysis to identify requirements whose removal causes architectural decisions to change:
  1. Calculate QA sensitivity by reducing each QA weight by a delta and re-running optimization.
  2. Rank QAs by the number of decisions that changed.
  3. Iteratively remove requirements contributing to the most sensitive QA.
  4. Re-run optimization after each removal; record changes as Architecturally Influential Sets (AIS).
- **SRS Trace:** §9.3
- **Legacy Trace:** FR-EXP-005, FR-EXP-006, FR-EXP-007
- **Priority:** Must
- **Acceptance Criteria:** Removing a group of 3 requirements changes the Database Management decision from SQL to NoSQL; this group is recorded as an AIS.

### 1.6 Output Protocol

#### FR-ARLO-019: Token-Optimized TOON Payload Output
- **Description:** ARLO SHALL produce a TOON (Trade-off Optimized Option Notation) payload containing:
  - `arlo_execution_metadata`: system name, optimization strategy, `arlo_reference` (format: `EXP-{SystemName}-{BatchID}`).
  - `alpha_decisions`: categories, selected patterns, weighted scores, satisfied QAs, primary driver IDs.
  - `beta_influential_sets`: sensitivity ranks, target QAs, impacted decisions, AIS requirement IDs.
  - `gamma_asr_registry`: total input requirements, total ASRs, ASR ID list, discarded ID list.
  All output relies on Requirement IDs rather than full text descriptions to optimize tokens.
- **SRS Trace:** §9.3
- **Legacy Trace:** FR-RS-001 (migration: text reports → TOON payload)
- **Priority:** Must
- **Acceptance Criteria:** The TOON payload validates against the SRS §9.3 schema; all IDs reference valid entries from the input CSV.

#### FR-ARLO-020: ARLO Reference ID in Output
- **Description:** The `arlo_reference` field (format: `EXP-{SystemName}-{BatchID}`) SHALL be explicitly included in the `arlo_execution_metadata` block. All downstream agents (RAA, AGA, SA) SHALL inherit this field directly from ARLO's output.
- **SRS Trace:** §9.3
- **Priority:** Must
- **Acceptance Criteria:** The RAA's TOON IR contains an `arlo_reference` field whose value matches the one in ARLO's output.

#### FR-ARLO-021: Read-Only Decision Output
- **Description:** ARLO's output decisions (Alpha Decisions, Beta Influential Sets, Gamma ASR Registry) SHALL be read-only. The system SHALL NOT allow users to modify individual ARLO decisions. The `interrupt_before` mechanism pauses for user review of the PDF report, not for decision modification.
- **SRS Trace:** §9.3, §1.7
- **Priority:** Must
- **Acceptance Criteria:** The ARLO Page displays decisions without edit controls; clicking "Next" resumes the pipeline without any decision modification capability.

#### FR-ARLO-022: LangGraph interrupt_before for User Review
- **Description:** After ARLO completes, LangGraph's `interrupt_before` mechanism SHALL pause the pipeline at the `arlo_user_review_node`. The system SHALL auto-generate the ARLO PDF report (Template A) and display it for user review. The user adds optional "Notes for AGA" and clicks "Next" to resume.
- **SRS Trace:** §10.3, §1.7
- **Priority:** Must
- **Acceptance Criteria:** The pipeline pauses after ARLO completion; the PDF report is displayed; clicking "Next" resumes from the checkpoint.

## 2. Non-Functional Requirements (NFR)

### NFR-ARLO-001: ILP Solver Timeout
- **Description:** The ILP solver SHALL have a configurable wall-clock timeout per batch (default: 120 seconds). Exceeding this limit triggers an automatic downgrade to Greedy for that batch only.
- **SRS Trace:** §13.2 (A-1)
- **Metric:** ILP optimization for a 7-category, 20-pattern matrix completes within 30 seconds.

### NFR-ARLO-002: LLM Memory Management
- **Description:** ARLO SHALL monitor the token count of its `ConversationSummaryBufferMemory`. If the buffer exceeds `max_input_tokens × 0.85`, forced memory summarization SHALL compress the buffer to 50% of max tokens.
- **SRS Trace:** §13.2 (A-3)
- **Metric:** Memory summarization completes within 5 seconds without data loss.

### NFR-ARLO-003: Retry Delegation
- **Description:** Network layer execution (circuit breaking, rate limiting, HTTP 429 management) SHALL be handled by the Orchestrator/LangGraph, not ARLO. ARLO SHALL be strictly focused on business logic.
- **SRS Trace:** §9.3
- **Metric:** ARLO's code contains no HTTP client or retry logic; all LLM calls go through LangChain Chat Models.

### NFR-ARLO-004: LLM Hallucination Handling for QA Classification
- **Description:** If the LLM produces an invalid QA (e.g., "SP" instead of "PE"), Pydantic validation SHALL trigger a retry with the valid QA enum explicitly injected. On 2nd failure, the requirement SHALL be assigned to a default "Unclassified" QA bucket and flagged for manual review.
- **SRS Trace:** §13.2 (A-2)
- **Metric:** Invalid QA detection triggers retry within 2 seconds; maximum 2 retries per requirement.

## 4. Disaster Recovery Requirements (DR)

### DR-ARLO-001: ILP Solver Timeout (A-1)
- **Failure Mode:** Solver wall-clock exceeds 120 seconds per batch.
- **Blast Radius:** Current optimization batch stalls; all subsequent batches blocked.
- **Recovery Action:** Downgrade from ILP to Greedy solver for the timed-out batch only. Log the downgrade. Mark batch as `optimization_method: greedy`.
- **User-Facing Message:** ⚠️ *"Optimization for batch {N} exceeded the time limit. A faster heuristic was used. Results may be slightly less optimal."*
- **SRS Trace:** §13.2 (A-1)

### DR-ARLO-002: LLM Hallucination — Invalid QA (A-2)
- **Failure Mode:** Pydantic `ValidationError` on QA response (e.g., "SP" instead of "PE").
- **Blast Radius:** Single requirement misclassified; sensitivity analysis may be skewed.
- **Recovery Action:** Retry with explicit constraint prompt (inject valid QA enum list). On 2nd failure, assign to default "Unclassified" QA bucket and flag for user review.
- **User-Facing Message:** ⚠️ *"ARLO could not classify requirement {REQ-ID}. It has been flagged for manual review."*
- **SRS Trace:** §13.2 (A-2)

### DR-ARLO-003: Memory Overflow (A-3)
- **Failure Mode:** Token count of ARLO's ConversationSummaryBufferMemory exceeds LLM's context window.
- **Blast Radius:** ARLO loses context → duplicate/contradictory grouping.
- **Recovery Action:** Forced memory summarization (compress buffer to 50% of max tokens). Log the event. If repeated, recommend larger-context LLM.
- **User-Facing Message:** ℹ️ *"ARLO's memory was optimized to continue processing. No action needed."*
- **SRS Trace:** §13.2 (A-3)
