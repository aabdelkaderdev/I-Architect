# PlantUML Rendering Server — Functional & Non-Functional Requirements

> **Version:** 1.0
> **Source Plan:** `requirements_planning/3_plantuml_server/plantuml_requirements_plan.md`
> **SRS Reference:** I-Architect SRS v2.3
> **Date:** 2026-03-24

## 1. Functional Requirements (FR)

### FR-PUML-001: SVG Rendering Endpoint
- **Description:** The system SHALL expose an HTTP endpoint at `POST /svg/` that accepts raw PlantUML code as the request body (content type `text/plain` or `application/x-www-form-urlencoded`) and returns the rendered diagram as raw SVG XML suitable for display in Streamlit UI components.
- **SRS Trace:** §12.2
- **Priority:** Must
- **Acceptance Criteria:** A valid `POST /svg/` request with correct PlantUML syntax returns HTTP `200` with `Content-Type: image/svg+xml` and a well-formed SVG document.

### FR-PUML-002: PNG Rendering Endpoint
- **Description:** The system SHALL expose an HTTP endpoint at `POST /png/` that accepts raw PlantUML code as the request body and returns the rendered diagram as binary PNG data suitable for embedding into PDF reports generated by WeasyPrint.
- **SRS Trace:** §12.2
- **Priority:** Must
- **Acceptance Criteria:** A valid `POST /png/` request with correct PlantUML syntax returns HTTP `200` with `Content-Type: image/png` and a valid PNG binary payload.

### FR-PUML-003: Rendering Timeout Policy
- **Description:** The PlantUML server SHALL support rendering tasks for complex C4 diagrams with a maximum timeout of 300 seconds (5 minutes). If a request exceeds 300 seconds, the Orchestrator SHALL terminate the connection and raise a `RenderTimeoutError`.
- **SRS Trace:** §12.2
- **Priority:** Must
- **Acceptance Criteria:** Rendering requests that exceed 300 seconds are terminated by the Orchestrator; a `RenderTimeoutError` is raised and logged.

### FR-PUML-004: Orchestrator-Side Render Cache (Redis)
- **Description:** The Orchestrator SHALL implement a caching layer before calling the PlantUML server. The cache key SHALL be the SHA-256 hash of the normalized (whitespace-trimmed) PlantUML code string. Cached artifacts SHALL be stored in Redis with the following structure:
  - `plantuml:svg:{hash}` → compressed SVG string
  - `plantuml:png:{hash}` → base64-encoded PNG string
  - **TTL:** 24 hours (auto-expiry).
- **SRS Trace:** §12.2
- **Priority:** Should
- **Acceptance Criteria:** Re-rendering identical PlantUML code returns the cached result without hitting the PlantUML server; cache entries expire after 24 hours.

### FR-PUML-005: Syntax Error Propagation (HTTP 400)
- **Description:** When the PlantUML server returns HTTP `400` (Bad Request) indicating invalid PlantUML syntax, the Orchestrator SHALL capture the error details from the `X-PlantUML-Error` header or response body (which includes line numbers) and feed this error text back to the AGA as a "Correction Prompt" to trigger the self-correction retry loop (SRS §13.4, G-1).
- **SRS Trace:** §12.2, §13.4
- **Priority:** Must
- **Acceptance Criteria:** An HTTP `400` response is intercepted by the Orchestrator; the exact error message including line numbers is passed to the AGA's self-correction prompt.

### FR-PUML-006: Server Crash Detection (HTTP 500)
- **Description:** When the PlantUML server returns HTTP `500` (Internal Server Error) indicating a JVM crash or OOM condition, the Orchestrator SHALL trigger a "Service Unhealthy" alert in the UI and SHALL NOT retry automatically indefinitely.
- **SRS Trace:** §12.2, §13.4
- **Priority:** Must
- **Acceptance Criteria:** An HTTP `500` response is surfaced to the user via a full-screen modal; the system logs the error and does not enter an infinite retry loop.

### FR-PUML-007: Input Sanitization (Security)
- **Description:** Before sending PlantUML code to the rendering server, the Orchestrator SHALL strip potentially malicious pragmas including `!include` directives that reference local file system paths (e.g., `!include file:///etc/passwd`) or external URLs. Only `!include` directives pointing to whitelisted internal standard libraries (e.g., C4-PlantUML stdlib) SHALL be permitted.
- **SRS Trace:** §12.2
- **Priority:** Must
- **Acceptance Criteria:** A PlantUML payload containing `!include file:///etc/passwd` is sanitized (the directive is removed or denied) before transmission; whitelisted includes (e.g., `!include https://raw.githubusercontent.com/plantuml-stdlib/C4-PlantUML/master/C4_Container.puml`) are preserved.

## 2. Non-Functional Requirements (NFR)

### NFR-PUML-001: Memory Constraint
- **Description:** The PlantUML server container SHALL operate within a hard memory limit of 512MB RAM as enforced by Docker Compose resource limits. JVM tuning parameters SHALL be set to `JAVA_OPTS="-Xmx400m -Xms128m -Djava.awt.headless=true"`, reserving ~112MB for the Jetty container overhead and OS processes.
- **SRS Trace:** §10.1
- **Metric:** `docker stats` reports peak memory usage ≤ 512MB under sustained rendering load.

### NFR-PUML-002: Concurrency Limit
- **Description:** Due to the single-threaded nature of Graphviz (used internally by PlantUML), the Orchestrator SHALL limit concurrent render requests to a maximum of 3 (matching the maximum number of parallel AGA agents in Workflow 3) to prevent Jetty thread pool exhaustion.
- **SRS Trace:** §10.1, §9.1
- **Metric:** Under 3 simultaneous render requests, all complete successfully within 300 seconds with no thread pool exhaustion errors.

### NFR-PUML-003: Stateless Operation
- **Description:** The PlantUML server container SHALL be completely stateless — no rendered artifacts, configuration, or intermediate data persists between requests or across container restarts.
- **SRS Trace:** §10.1
- **Metric:** Container restart has zero effect on subsequent rendering requests; no persistent volumes are mounted.

### NFR-PUML-004: Startup Health Check
- **Description:** The Orchestrator SHALL perform a startup health check (`GET /` or `HEAD /`) against the PlantUML server before marking the pipeline as "Ready." Pipeline execution SHALL be blocked until the health check passes.
- **SRS Trace:** §10.1
- **Metric:** Pipeline startup is blocked until the PlantUML server responds with HTTP `200` to the health check request.

### NFR-PUML-005: Automatic Restart Policy
- **Description:** The Docker Compose configuration SHALL set the PlantUML server restart policy to `restart: on-failure` to recover from JVM crashes without manual intervention.
- **SRS Trace:** §10.1
- **Metric:** After a simulated JVM crash (OOM kill), the container automatically restarts and passes health check within 30 seconds.

## 3. Interface Requirements (IR)

### IR-PUML-001: Orchestrator → PlantUML Rendering Interface
- **Source:** Orchestrator Container (Django/LangGraph)
- **Target:** PlantUML Server Container
- **Protocol:** HTTP REST
- **Endpoints:**
  | Method | Path | Request Body | Response |
  |:--|:--|:--|:--|
  | `POST` | `/svg/` | Raw PlantUML code (`text/plain`) | SVG XML (`image/svg+xml`) |
  | `POST` | `/png/` | Raw PlantUML code (`text/plain`) | PNG binary (`image/png`) |
  | `GET` | `/` | — | Health check `200 OK` |
- **Error Codes:**
  | HTTP Code | Meaning | Orchestrator Action |
  |:--|:--|:--|
  | `200` | Render successful | Cache and return result |
  | `400` | Syntax error | Extract error → AGA correction prompt |
  | `500` | Server crash/OOM | UI alert, log error, no retry loop |

### IR-PUML-002: Redis Cache Interface
- **Purpose:** Orchestrator-side rendering cache to avoid redundant PlantUML server calls.
- **Key Schema:**
  - `plantuml:svg:{sha256_hash}` → Compressed SVG string
  - `plantuml:png:{sha256_hash}` → Base64-encoded PNG string
- **TTL:** 24 hours

### IR-PUML-003: Docker Network Configuration
- **Network:** `i-architect-net` (bridge)
- **Service Name:** `plantuml-server`
- **Image:** `plantuml/plantuml-server:jetty-v1.2026.2`
- **Internal Port:** `8080` (Jetty default)
- **Host Port:** `8300`
- **Resource Limits:** `mem_limit: 512M`

## 4. Disaster Recovery Requirements (DR)

### DR-PUML-001: PlantUML Syntax Error Recovery (G-1)
- **Failure Mode:** AGA generates PlantUML code with syntax errors (missing `@startuml`/`@enduml`, unclosed brackets, invalid macros).
- **Trigger:** PlantUML server returns HTTP `400`.
- **Recovery Action:** The Orchestrator feeds the exact syntax error message back to the AGA as a correction prompt. The AGA re-invokes the LLM with the error context (up to 2 retries after the initial attempt, total 3 attempts). On 3rd failure, a minimal valid stub with comment `' GENERATION_FAILED: Manual review required` is output.
- **User-Facing Message:** ⚠️ *"The diagram could not be generated correctly after multiple attempts. A placeholder has been saved."*
- **SRS Trace:** §13.4 (G-1)

### DR-PUML-002: PlantUML Server Crash Recovery
- **Failure Mode:** JVM crash or OOM kill within the PlantUML container.
- **Trigger:** HTTP `500` or container exit code `137`.
- **Recovery Action:** Docker's `restart: on-failure` policy automatically restarts the container. The Orchestrator displays a "Service Unhealthy" alert and retries the render request after a 10-second delay. If the container fails to restart within 60 seconds, the UI displays a blocking error modal.
- **User-Facing Message:** 🛑 *"The diagram rendering service crashed and is restarting. Please wait..."*
- **SRS Trace:** §13

### DR-PUML-003: Rendering Timeout Recovery
- **Failure Mode:** Complex C4 diagram rendering exceeds 300-second timeout.
- **Trigger:** Orchestrator-side timeout fires.
- **Recovery Action:** The Orchestrator terminates the HTTP connection, raises `RenderTimeoutError`, and suggests simplifying the diagram or splitting it into sub-diagrams.
- **User-Facing Message:** ⚠️ *"Diagram rendering timed out after 5 minutes. Consider simplifying the architecture or splitting into sub-diagrams."*
- **SRS Trace:** §12.2
