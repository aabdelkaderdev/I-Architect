# Research Notes: RAA Failure Mode Mitigations

This document outlines the design decisions and technical rationale for implementing the failure mode mitigations in the RAA pipeline, complying with RAA_Plan.md Section 18 and Section 22H.

---

## 1. Startup Embedding DB Verification and Corruption Rebuilds

### Decision
- **ASR DB Verification**: The `prepare_embeddings` node will attempt to connect to `asr_embeddings.db`. If the database is missing or the connection raises `sqlite3.DatabaseError`, the node halts execution and raises a blocking `RuntimeError`. If the database exists but lacks any ASR requirement ID from the input state, a blocking `RuntimeError` is raised.
- **Non-ASR DB Automatic Rebuild**: If `non_asr_embeddings.db` is missing, the node automatically creates it. If opening or querying `non_asr_embeddings.db` raises a `sqlite3.DatabaseError` (indicating corruption or database locking), the node catches the error, logs a prominent `WARNING`, unlinks/deletes the corrupted database file, establishes a fresh connection to recreate the schema, and executes a full rebuild from scratch.

### Rationale
- ASR embeddings are produced by ARLO. Since RAA cannot regenerate them, a missing or corrupted ASR database is a terminal failure requiring operator action (running ARLO).
- Non-ASR embeddings are fully owned and generated by RAA from raw requirements text. They can be safely rebuilt automatically without operator intervention, maximizing pipeline resilience.

---

## 2. Embedding Cache Integrity and Stale Hash Detection

### Decision
- The `prepare_embeddings` node will read all existing cached `requirement_id` and `text_hash` values from `non_asr_embeddings.db`.
- For each non-ASR requirement in the input state, its current SHA-256 text hash is computed.
- If a requirement ID exists in the cache but its cached hash does not match the newly computed hash:
  - Log a `WARNING` indicating a stale embedding has been detected for that specific requirement.
  - Recompute the embedding for that requirement only.
  - Persist the new embedding, text hash, and model name to the database.

### Rationale
- Avoids silent reuse of stale embeddings when requirement descriptions are modified.
- Avoids the overhead of rebuilding the entire cache, only recomputing modified elements (O(n) where n is the number of changed requirements).

---

## 3. SQLite Concurrency and WAL Mode

### Decision
- Enable WAL (Write-Ahead Logging) mode on all database connections: `asr_embeddings.db`, `non_asr_embeddings.db`, and `raa_graph.db` (checkpointer).
- Execute `PRAGMA journal_mode=WAL;` immediately upon opening any connection.
- In downstream nodes that only require read access (`batch_construction`, `overlap_bridging`, etc.), open SQLite connections in read-only mode using a URI string: `sqlite3.connect("file:{db_path}?mode=ro", uri=True)`.

### Rationale
- WAL mode allows concurrent readers to access the database without being blocked by writers.
- Read-only connections prevent accidental writes, minimize file locking overhead, and allow multiple parallel LangGraph subgraphs to execute query operations safely in parallel.

---

## 4. Desync Recovery during Final Merge

### Decision
- The `final_merge` node will perform a desync verification step by checking that every batch index in `range(len(batch_queue))` is present in the `best_batch_output` state dictionary.
- If any key is missing or maps to a corrupted/empty fragment:
  - Log a warning indicating a desync has been detected.
  - Set the state channel `batch_cursor` to the first missing batch index.
  - Raise a recoverable `ValueError` or transition back to the batch execution step (once the graph execution loop is fully wired).
  - This allows the pipeline orchestrator/runner to detect the desync and re-run the missing batch, saving prior progress while ensuring the final merged model is complete.

### Rationale
- Prevents generating a partial or corrupt C4 model when individual batch executions fail to write to the checkpointer or get dropped during network interruptions.
- Provides a clean fallback path that allows automatic or operator-guided targeted re-runs.

---

## 5. Failure Modes Register Integration (Section 18 & 22H)

### Decision
Update the project's failure modes register to map the following Section 22H risks:
1. **Process killed mid-pipeline**: Mitigated by SQLite checkpointing (`batch_cursor` resumption).
2. **Checkpoint DB unavailable/corrupt**: Mitigated by corrupt fallback (rename to `.corrupted` + fresh start).
3. **Batch cursor desync**: Mitigated by final merge keys verification and targeted batch re-runs.
4. **Embedding DB missing/corrupt**: Mitigated by startup verification (ASR DB) and automatic rebuild (Non-ASR DB).
5. **SQLite DB locking**: Mitigated by WAL mode and read-only URI connections.

### Rationale
- Provides a centralized audit trail of pipeline failure modes and guarantees that every identified risk has a validated mitigation.
