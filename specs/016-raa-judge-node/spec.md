# Feature Specification: RAA Per-Batch Judge Node

**Feature Branch**: `002-raa-subgraph`

**Created**: 2026-05-19

**Status**: Draft

**Input**: `RAA_Plan.md` Section 13

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Evaluate parallel subgraphs and merge architectural outputs (Priority: P1)

At the end of each batch's execution, the pipeline must reconcile the independent architectural proposals generated by the three parallel strategy subgraphs (RAA-A, RAA-B, RAA-C). The Judge node receives these proposals, scores them against the batch's requirements using SAAM scenarios, selects the strongest primary fragment, and then uses a deterministic 4-step algorithm to merge useful, non-conflicting components from the losing fragments (residual scan). Finally, it assembles the entities into a hierarchical tree, flags unresolved conflicts, updates the `running_arch_model`, and advances the `batch_cursor` to process the next batch.

**Why this priority**: The judge node is the consensus mechanism of the RAA pipeline. Without it, the three parallel subgraphs would produce contradictory overlaps, preventing the construction of a cohesive, single architecture model.

**Independent Test**: Write a unit test that mocks three parallel subgraph `ArchFragment` outputs containing both identical entities and deliberate conflicts (e.g. mismatched parent IDs). Execute the judge node and verify that: 
- Identical entities are deduplicated.
- Conflicting parent IDs produce `hierarchy_conflict` entries in `open_questions`.
- Orphan components from losing fragments are rejected during union and produce `coverage_gap` entries.
- The `batch_cursor` is incremented.

**Acceptance Scenarios**:

1. **Given** three parallel fragments, **When** evaluated, **Then** the judge scores them, selects the best, and deduplicates identical entities.
2. **Given** fragments proposing the same container ID but different `parent_system_id` values, **When** merged, **Then** a `hierarchy_conflict` is recorded in `open_questions`.
3. **Given** a residual component in a losing fragment whose parent container is absent from the primary selection, **When** the residual scan runs, **Then** the component is omitted and a `coverage_gap` is recorded.
4. **Given** successful merge completion, **When** finalized, **Then** the `running_arch_model` receives the newly assembled hierarchical tree, and `batch_cursor` increments by 1.

---

### Edge Cases

- **Reduced Confidence Scoring**: If the batch is flagged with `reduced_confidence = true` (by the coherence gate), the judge node applies a 0.5× multiplier to all SAAM scores, reducing its overall weight in the final system architecture.
- **Scope Mismatches**: If two fragments define the same relationship but assign different diagram scopes, the judge selects the scope consistent with endpoint types and records a `scope_conflict`.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST implement the judge node under `raa/nodes/judge.py`.
- **FR-002**: System MUST use `llm_judge` from the runtime configuration context to perform SAAM scenario scoring on the three candidate fragments.
- **FR-003**: System MUST execute a 4-step deterministic merge: Entity deduplication, Relationship deduplication, Coverage union with orphan prevention, and Tree assembly.
- **FR-004**: System MUST record hierarchy conflicts (`parent_system_id` or `parent_container_id` mismatches) and scope conflicts as structured entries in the `open_questions` state.
- **FR-005**: System MUST perform a residual scan on losing fragments to add non-conflicting entities/relationships possessing positive SAAM coverage into the final selection.
- **FR-006**: System MUST reject orphaned components or containers during the residual union step and log them as a `coverage_gap` in `open_questions`.
- **FR-007**: System MUST assemble the final batch entities into a nested hierarchical C4 tree structure prior to updating the `running_arch_model`.
- **FR-008**: System MUST increment `batch_cursor` at the completion of the node to advance graph execution.

### Key Entities *(include if feature involves data)*

- **ArchFragment**: The data structure representing partial C4 entities proposed by a strategy subgraph.
- **OpenQuestion**: A state tracking structure containing conflict records (e.g. `hierarchy_conflict`, `scope_conflict`, `coverage_gap`).
- **Batch Cursor**: The integer state counter driving batch traversal.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: The deterministic merge algorithm successfully deduplicates identical entities without generating orphan components/containers.
- **SC-002**: All encountered conflicts and coverage gaps are accurately captured in the `open_questions` state structure.
- **SC-003**: The LangGraph pipeline correctly advances to the next batch upon node completion via the `batch_cursor` update.
