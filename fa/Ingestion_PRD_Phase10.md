# Data Ingestion & Requirement Filtering — Phase 10: Orchestrator Interface, Directory Layout & Design Principles

## Summary

This phase defines the boundary between the ingestion pipeline and the orchestrator — what the orchestrator provides, how it invokes the pipeline, and what it receives back. It also specifies the full project directory layout for the `ingestion/` code package and the `Skills/Ingestion/` resource bundle, and the design principles that govern the module. This is the integration and handoff spec.

**Depends on:** All prior phases. The orchestrator interface references the state schema (Phase 9), configuration (Phase 2), and output contract (Phase 1). The directory layout references every extractor (Phases 4–6), the normaliser (Phase 6), the RFA (Phases 7–8), and the prompt template (Phase 7).

**Required reading before:** nothing. This is the final phase.

## 1. Orchestrator Interface

### 1.1 What the Orchestrator Provides

Before invoking the ingestion graph, the orchestrator must provide:

| Item | Description |
|------|-------------|
| `file_path` | Absolute path to the uploaded file, written to `raw_file_path` state channel |
| `ingestion_config` | Serialised `IngestionConfig` (Phase 2, §3), or `None` to use all defaults |
| `filter_config` | Serialised `FilterConfig` (Phase 2, §4), or `None` to use all defaults |
| `llm` | A `ChatOpenAI` (or compatible `BaseChatModel`) instance, passed via LangGraph runtime `context` |
| `db_path` | Absolute path to the checkpoint database (e.g., `projects/{name}/checkpoints/ingestion.db`), passed to the compile function |
| `thread_id` | A unique thread identifier for checkpointing, derived by the orchestrator using the formula in Phase 9, §6.3 |

### 1.2 How the Orchestrator Invokes the Pipeline

The orchestrator calls the ingestion module's compile function with the checkpoint database path, then invokes the compiled graph with the input state and the LLM in context:

```python
graph = compile_ingestion_graph(db_path=db_path)
result = graph.invoke(
    {
        "raw_file_path": file_path,
        "ingestion_config": ingestion_config,
        "filter_config": filter_config,
    },
    context={"llm": llm_instance},
    config={"configurable": {"thread_id": thread_id}},
)
```

The orchestrator does not import or instantiate any ingestion node directly. It interacts only with the compiled graph and the input state channels.

### 1.3 What the Orchestrator Receives Back

The graph returns the final state dict. The orchestrator reads:

| Channel | Type | What the Orchestrator Does With It |
|---------|------|-----------------------------------|
| `extracted_requirements` | `dict[str, str]` | Passes the value to `ARLOInput.requirements` — this is the handoff to ARLO |
| `filter_report` | `dict` or `null` | Stores for downstream audit and includes in the final PDF report generated by the reporting agent |

### 1.4 What the Orchestrator Passes Through

The following channels exist in the broader pipeline state and are passed through the ingestion graph unchanged. The orchestrator is responsible for preserving them:

| Channel | Handled By |
|---------|------------|
| `requirements` | Receives the value of `extracted_requirements` at the ARLO boundary |
| `experiment_config` | Passed through to ARLO unchanged |
| `matrix` | Passed through to ARLO unchanged |

### 1.5 Orchestrator Responsibilities

The orchestrator owns these responsibilities — the ingestion pipeline does not handle them:

- **Directory creation:** The orchestrator creates the `projects/{name}/checkpoints/` directory before invoking the compile function. The ingestion module assumes the directory exists.
- **Config validation:** The orchestrator validates `IngestionConfig` and `FilterConfig` against the rules in Phase 2, §6 before passing them to the graph.
- **File existence check:** The orchestrator verifies the file exists at `file_path` before graph invocation.
- **Error handling:** The orchestrator catches all exceptions from the ingestion graph (Phase 1, §8) and decides how to surface them to the user.
- **Checkpoint lifecycle:** The orchestrator archives or prunes checkpoint databases according to the 7-day retention policy.

## 2. Directory Layout

### 2.1 Code Package: `ingestion/`

Runtime code and prompt templates follow the same convention as `arlo/`, `raa/`, `aga/`, and `sa/`:

```
ingestion/
├── __init__.py
├── config.py                  # IngestionConfig, FilterConfig dataclasses
├── router.py                  # Format detection and routing (Phase 3)
├── extractors/
│   ├── __init__.py
│   ├── base.py                # Abstract base extractor
│   ├── pdf_extractor.py       # PDF → text blocks (Phase 4)
│   ├── docx_extractor.py      # DOCX → text blocks (Phase 5)
│   ├── txt_extractor.py       # TXT → text blocks (Phase 5)
│   └── json_validator.py      # JSON schema validation + passthrough (Phase 6)
├── normaliser.py              # ID assignment, dedup, length filtering (Phase 6)
├── filter_agent.py            # Requirement Filtering Agent — LLM-based (Phases 7–8)
├── exceptions.py              # Custom exceptions (Phase 1, §8)
├── models.py                  # Pydantic models: FilteredRequirement, FilterBatch (Phase 7, §6)
└── prompts/
    └── filter_classification.md  # Runtime prompt for RFA LLM call (Phase 7, §5)
```

### 2.2 Skills Resource Bundle: `Skills/Ingestion/`

Skill definitions and reference documents only — never runtime code or prompt templates:

```
Skills/Ingestion/
├── SKILL.MD                    # Ingestion pipeline skill definition
└── references/
    └── Signal_Noise_Taxonomy.md  # Normative classification criteria (Phase 7, §3–4)
```

The `Skills/Ingestion/` directory is for skill metadata and methodology references. The prompt template that is loaded at runtime lives in `ingestion/prompts/`, not here. This separation follows the same convention used by `Skills/ARLO/`, `Skills/RAA/`, `Skills/AGA/`, and `Skills/SA/`.

### 2.3 Checkpoint Storage

Checkpoint databases are project-scoped. The directory is created and managed by the orchestrator:

```
projects/{project_name}/checkpoints/
├── orchestrator.db
├── ingestion.db
├── arlo.db
├── raa_graph.db
├── aga.db
├── sa.db
└── rga.db
```

The ingestion module writes only to `ingestion.db` at the path provided by the orchestrator.

## 3. Design Principles

### 1. Two distinct stages, one module.

Data Ingestion (deterministic) and Requirement Filtering (LLM-based) are fundamentally different operations. They share a module because they share a goal — producing a clean `dict[str, str]` — but their internals must not bleed together. Extraction code never calls the LLM. Filtering code never parses a file.

### 2. Library-native over abstraction.

The pipeline uses `pdfplumber`, `python-docx`, and `chardet` directly — not LangChain's document loaders. Fine-grained extraction control matters more than a uniform abstraction. Each extractor owns its library's behaviour completely.

### 3. Fail loud, fail early.

The pipeline validates at every boundary: file size before parsing, schema before passthrough, requirement count before handoff. Silent coercion — converting a non-compliant JSON into a compliant one — is explicitly rejected. The user must fix bad input.

### 4. Deterministic where possible.

Only the RFA classification (Signal vs Noise) uses an LLM. Format detection, extraction, normalisation, threshold application, and report generation are all deterministic Python. This keeps the pipeline reproducible and testable at every non-LLM step.

### 5. Configuration flows one way.

The orchestrator writes config once. Nodes read it. No node modifies configuration. No node reads configuration from anywhere other than state. This prevents config drift mid-pipeline.

### 6. LLM lives in context, not state.

The LLM instance is passed via LangGraph's runtime `context`. It never appears in a state channel. State remains serialisable, and the LLM cannot accidentally be included in checkpoint snapshots.

### 7. Orchestrator owns the boundary.

The ingestion pipeline does not create directories, validate config, check file existence, or manage checkpoint lifecycles. The orchestrator handles all infrastructure concerns. The pipeline assumes its inputs are valid and its dependencies exist.

### 8. Passthrough for clean input.

A compliant JSON file that already matches the standard format should cost as close to zero as possible to process. No extraction, no normalisation, and by default no filtering. The pipeline gets out of the way when there is nothing to do.

## 4. Downstream Consumers

The ingestion pipeline's output feeds into:

| Consumer | What It Receives |
|----------|-----------------|
| ARLO | `extracted_requirements` (`dict[str, str]`) — the validated requirement set, handed off by the orchestrator via `ARLOInput.requirements` |
| RAA | (downstream of ARLO) — the architecture model is built from the requirements ARLO processes |
| AGA | (downstream of RAA) — the PlantUML diagrams are generated from the architecture model |
| SA | (downstream of AGA) — the scoring agent evaluates the architecture against the original requirements |

The ingestion pipeline has no direct dependency on any of these modules. It produces requirements and exits.
