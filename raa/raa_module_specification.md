# RAA Module Specification

**Version**: 1.2 — Human Review Gate Added  
**Date**: 2026-05-21  
**Source Authority**: RAA_Plan.md v3, SAAM Audit Report  
**Scope**: Conceptual architecture and requirements only — no implementation details or code

> [!IMPORTANT]
> This specification assumes the **older ARLO version** (commit `3cf20c6`) which does **not** persist ASR embeddings to SQLite and provides `non_asr` as a flat list of requirement IDs (`list[str]`), not full dicts. RAA therefore owns **all** embedding generation — both ASR and non-ASR.

---

## 1. Purpose and Pipeline Position

### 1.1 What RAA Does

The **Requirement Analysis Agent (RAA)** transforms a classified set of software requirements into a structured, hierarchical architecture model conforming to the C4 modelling standard. It is the **middle stage** of a three-agent pipeline:

| Stage | Agent | Responsibility |
|---|---|---|
| Upstream | **ARLO** (Architecturally-significant Requirements Locator & Organizer) | Classifies requirements, identifies quality attributes, clusters them into condition groups. Generates embeddings internally for its own clustering but does **not** persist them for downstream use |
| **Current** | **RAA** | Partitions requirements into batches, runs parallel architecture analysis strategies, judges and merges results into a single coherent C4 model |
| Downstream | **AGA** (Architecture Generation Agent) | Renders the C4 JSON model into PlantUML diagrams and code artifacts |

### 1.2 What RAA Produces

RAA's sole output is a **C4-compliant JSON model** — a hierarchical entity-relationship structure describing systems, containers, components, persons, external systems, their relationships, selected architectural patterns, a diagram manifest for AGA, and confidence metadata. RAA produces **no diagrams, no code, and no visual artifacts**.

### 1.3 Core Design Principles

1. **Batch-sequential, strategy-parallel**: Requirements are processed in ordered batches (sequentially). Within each batch, three independent analysis strategies run in parallel and are reconciled by a judge.
2. **Incremental coherence**: Each batch builds upon the accumulated model from all previous batches. Later batches receive the running model as a hard constraint.
3. **LLM isolation**: LLM instances are runtime-injected, never serialised into persistent state. Four distinct LLM slots allow heterogeneous model assignments.
4. **Embedding persistence outside state**: Dense vector embeddings are generated by RAA and stored in SQLite databases on disk, not in the graph's state channels, keeping checkpoint snapshots lean. ARLO does not provide embeddings — RAA generates all of them.
5. **Deterministic where possible**: Merge, deduplication, tree assembly, and manifest generation are deterministic algorithms. LLMs are used only for entity/relationship extraction, pattern selection, and conflict reconciliation.

## 2. Inputs

### 2.1 From ARLO (Upstream Agent)

| Input | Type | Description |
|---|---|---|
| **ASRs** | `list[dict]` | Architecturally significant requirements — each dict carrying an integer `id`, `is_architecturally_significant` (bool), `quality_attributes` (list of strings), and `condition_text` (string). The requirement description text is **not** included — it must be resolved from the original requirements input |
| **Non-ASRs** | `list[str]` | Requirement IDs **only** (not full dicts). These are string keys like `"R5"` identifying requirements that ARLO classified as non-architecturally-significant. No quality attributes or condition text are provided — RAA must resolve all metadata from the parent pipeline's requirements dict |
| **Condition Groups** | `list[dict]` | Clusters of ASRs sharing equivalent conditions, anchored by K-Means clustering over ASR embeddings. Each group has a `nominal_condition` (string), `conditions` (list of ASR indices), `requirements` (list of full ASR dicts), and `cluster` (int). Note: ARLO computes embeddings internally for this clustering but does **not** expose or persist them |
| **Quality Weights** | `dict[str, int]` | Aggregate quality attribute frequency counts across all satisfiable groups, used as relative weights for SAAM scoring |

> [!WARNING]
> The older ARLO version provides `non_asr` as `list[str]` (bare IDs), not `list[dict]`. RAA's normalisation step must enrich these IDs into full requirement records by looking up the description text from the parent pipeline's `requirements` dict and defaulting `quality_attributes` to an empty list and `condition_text` to null.

### 2.2 From the Parent Pipeline

| Input | Description |
|---|---|
| **Requirements** | The original full requirement set (ID → description text), used to resolve description text for both ASR and non-ASR entries during normalisation |

### 2.3 Embedding Persistence (SQLite — Owned by RAA)

The older ARLO version does **not** persist embeddings to disk. ARLO generates embeddings internally (in its state channel `embeddings: list[list[float]]`) for K-Means clustering, but these embeddings are not included in `ARLOOutput` and are not written to any external database.

**RAA is therefore the sole owner of all embedding persistence.** During its preparation phase, RAA generates and persists embeddings for **both** ASR and non-ASR requirements:

| Database | Written By | Contains |
|---|---|---|
| **ASR Embeddings** | RAA (during preparation) | 1024-dimensional dense vectors for all ASR condition texts |
| **Non-ASR Embeddings** | RAA (during preparation) | 1024-dimensional dense vectors for all non-ASR requirement description texts |

Both databases share an identical schema: requirement ID (primary key), embedding vector (binary blob), text hash (for cache integrity), and model name (for reproducibility).

### 2.4 Runtime Injection (Not State)

| Injection | Description |
|---|---|
| **4 LLM Instances** | One per functional role (RAA-A, RAA-B, RAA-C, Judge) — injected via runtime context, never serialised |
| **Checkpoint DB Path** | Orchestrator-provided path for SQLite checkpointing |
| **Output Directory** | Orchestrator-provided path for the final JSON model |

## 3. Output

### 3.1 Final Architecture Model

A single JSON file (`arch_model.json`) containing:

| Field | Description |
|---|---|
| **Systems** | All software systems under design, each containing its containers (which in turn contain their components). Relationships are nested at each level. |
| **Persons** | Global flat list of human actors, deduplicated across all batches |
| **External Systems** | Global flat list of external software systems, deduplicated across all batches |
| **Patterns** | Selected architectural patterns with rationales and quality attribute mappings |
| **Diagram Manifest** | Deterministically generated work queue for AGA — one entry per diagram to produce (context + container per system, component per container) |
| **Confidence Metadata** | Per-entity confidence flags including reduced-confidence markers for entities from incoherent batches |
| **Open Questions** | Unresolved conflicts, hotspots, ties, and coverage gaps for human review |

### 3.2 Hierarchy Rules

The model is **strictly hierarchical**: entity type is determined by structural position, not a discriminator field.

- A **System** is a top-level entry in the `systems` array
- A **Container** is nested inside a system's `containers` array
- A **Component** is nested inside a container's `components` array
- **Persons** and **External Systems** are flat, global, and referenced by ID from relationships

### 3.3 Diagram Manifest Contract

The manifest deterministically specifies: one **context diagram** per system, one **container diagram** per system, and one **component diagram** per container. Total diagrams = `(2 × number of systems) + (total containers across all systems)`. AGA traverses this manifest as a work queue without any independent filtering logic.

## 4. State Schema

### 4.1 Inner Working State (Graph Channels)

These channels exist only during RAA execution and are persisted via SQLite checkpointing for crash recovery.

| Channel | Purpose | Writer(s) |
|---|---|---|
| **Batch Queue** | Ordered list of requirement batches, each containing ASR IDs, non-ASR candidates, group metadata, similarity scores, and coherence flags | Batch construction (once) |
| **Batch Cursor** | Index of the current batch being processed — the authoritative resume marker | Judge (after each batch) |
| **Batch Outputs** | Raw ArchFragment outputs from each parallel subgraph, keyed by batch index. Must use an append-merge reducer since three subgraphs write concurrently | RAA-A, RAA-B, RAA-C (parallel) |
| **Best Batch Output** | Judge-selected and merged fragment per batch | Judge (one per batch) |
| **Running Architecture Model** | Accumulated hierarchical model built incrementally — injected into subsequent batch prompts as hard constraints | Judge (after each batch) |
| **Open Questions** | Unresolved conflicts flagged during merge — uses append reducer since multiple batches contribute | Judge (append per batch) |
| **Bridge Requirements** | Map of adjacent group pairs to shared bridge requirement IDs | Overlap bridging (once) |
| **Incoherent Batches** | Records of batches that failed the coherence gate, carrying reduced-confidence flags | Coherence gate (append) |
| **Embeddings Ready** | Boolean flag set after all embeddings are verified/generated — gates downstream nodes | Preparation (once) |
| **Batch Ordering Strategy** | Which ordering heuristic was used (risk-first, ASR count, or quality weight) | Batch queue ordering (once) |
| **Unprocessed Requirements** | Full normalised records of all requirements that were not assigned to any batch (neither as a primary ASR nor as a non-ASR candidate). Computed once after Phase 5 by diffing the full normalised requirement set against all requirement IDs present in the batch queue. Used exclusively by the Phase 8 residual pass | Batch queue ordering (once) |
| **Human Review Payload** | Read-only snapshot emitted to the caller at the Phase 7 interrupt point. Contains all accumulated open questions classified by resolution owner, the current merged model summary, and per-question context. Written once at the start of Phase 7; never mutated after emission | Phase 7 Human Review Gate (once) |
| **Human Answers** | Map from open question ID to human-provided resolution text, injected by the caller after an interrupt. Empty dict if `review_mode = autonomous` or if the timeout lapses with no input. Uses overwrite semantics (single write per run) | External caller (via interrupt resume) |
| **Review Mode** | Enum — `interactive` or `autonomous`. Injected at pipeline invocation. Controls whether Phase 7 issues a LangGraph interrupt or passes through immediately | Runtime injection |

### 4.2 Intermediate Data Types

**ArchFragment** — the partial output from a single subgraph run. Uses a semi-flat structure with explicit parent ID fields (containers carry `parent_system_id`, components carry `parent_container_id`) to simplify inter-fragment deduplication. The judge converts fragments into the fully nested ArchModel after merge.

**Batch** — a unit of work containing: the requirement IDs in this batch, full normalised requirement payloads, the group centroid vector, similarity scores per requirement, non-ASR candidate records, coherence score, split status, and sorting metadata.

**OpenQuestion** — a flagged conflict or gap. Fields: `id` (unique string, used as the key in `human_answers`), `type` (one of `change_risk`, `high_coupling`, `contention`, `tie`, `coverage_gap`, `hierarchy_conflict`, `scope_conflict`), `description` (human-readable summary), `context` (relevant requirement IDs and entity IDs), `resolution_owner` (`human_preferred` or `judge_resolvable`), `resolution` (null until resolved — either the human answer text or the judge's assumption text), and `assumption_flag` (boolean, true when the judge resolved without human input).

**IncoherentBatchRecord** — records a batch that failed the coherence gate: batch ID, coherence score, and `reduced_confidence = true`.

### 4.3 Reducer Requirements

Channels written by multiple nodes in the same execution step **must** use merge reducers to prevent data loss:

- **Batch Outputs**: dict-merge with list concatenation per batch key (three parallel writers)
- **Open Questions**: list append (multiple batches contribute)
- **Incoherent Batches**: list append

Single-writer channels use standard overwrite semantics.

## 5. Pipeline Phases

The RAA pipeline consists of **eight sequential phases**, with parallelism only within Phase 6. Phase 7 is the optional human review gate — it either pauses for human input or passes through immediately in autonomous mode.

### Phase 1 — Preparation

**Responsibility**: Normalise all requirements into a shared schema and generate all embeddings.

- **ASR Normalisation**: Convert ARLO's integer IDs to string keys (e.g., `1` → `"R1"`), resolve description text from the parent pipeline's requirements dict, rename `is_architecturally_significant` to `is_asr`, default missing fields
- **Non-ASR Enrichment**: ARLO provides `non_asr` as a flat list of requirement ID strings. RAA must enrich each ID into a full normalised record by resolving the description text from the parent pipeline's `requirements` dict, and setting `is_asr = false`, `quality_attributes = []`, `condition_text = null`
- **ASR Embedding Generation**: Generate 1024-dimensional dense vectors for all ASR condition texts and persist to the ASR embeddings database. Since the older ARLO does not persist embeddings externally, RAA must generate these from scratch using the same embedding model that ARLO uses internally (`mixedbread-ai/mxbai-embed-large-v1` via FastEmbed)
- **Non-ASR Embedding Generation**: Embed all non-ASR requirement description texts using the same model and persist to the non-ASR embeddings database
- **Cache Behaviour**: Check text hashes for each requirement — only re-embed entries with missing or stale hashes. This enables efficient re-runs without redundant computation
- **Gate**: Set `embeddings_ready = true` when complete. All downstream nodes gate on this flag

### Phase 2 — Batch Construction

**Responsibility**: For each ARLO condition group, assemble a batch of requirements by combining the group's ASRs with semantically similar non-ASRs.

- **Anchor Computation**: Calculate the centroid vector of all ASR embeddings in the group (loaded from the RAA-generated ASR embeddings database by requirement ID)
- **Non-ASR Selection**: Run approximate nearest-neighbour search against the RAA-generated non-ASR embeddings using the group centroid. Apply similarity threshold ≥ 0.65. Cap at 10 non-ASR candidates per group
- **Batch Assembly**: Batch = all ASRs in the condition group + selected non-ASR candidates. Store with group ID, centroid, and per-requirement similarity scores

### Phase 3 — Overlap Bridging

**Responsibility**: Ensure cross-batch coherence by injecting 1–3 shared "bridge" requirements into adjacent batches.

- Identify adjacent or related condition groups (same cluster ID or high centroid similarity)
- For each adjacent pair, find non-ASRs with high similarity to **both** group centroids
- Select 1–3 bridge requirements (hard cap: 3) and inject into both batches
- Record bridge mappings in state

### Phase 4 — Coherence Gate

**Responsibility**: Verify that each batch is semantically cohesive enough for reliable architecture analysis.

- Compute the average intra-batch cosine similarity between each requirement embedding and the batch centroid
- **If coherence score ≥ 0.55**: batch passes unchanged
- **If coherence score < 0.55**: attempt to split into 2 sub-clusters. If both sub-clusters pass, replace the batch with two sub-batches. If still incoherent after splitting, mark with `reduced_confidence = true` and run as a single-strategy batch (not three-parallel)

### Phase 5 — Batch Queue Ordering

**Responsibility**: Order batches by risk priority to ensure the most consequential architectural decisions are made earliest.

- **Default (Risk-First)**: Security and reliability batches precede performance and usability — high-risk decisions are made while the model is sparse and flexible
- **Alternatives**: By ASR count (largest groups first) or by aggregate quality weight
- Configurable via pipeline parameter

### Phase 6 — Execution Loop (Per-Batch)

**Responsibility**: For each batch, run parallel architecture analysis strategies and reconcile results.

This phase loops through the batch queue sequentially. For each batch:

#### 6a. Fan-Out (Parallel Subgraph Dispatch)

Dispatch three parallel subgraph runs (or one for reduced-confidence batches). Each receives:
- The current batch's normalised requirements
- Quality weights from ARLO
- The current running architecture model as a hard constraint
- Bridge requirements context
- Relevant reference document excerpts
- Its assigned LLM instance

#### 6b. Subgraph Execution (Three Parallel Strategies)

Each subgraph produces an **ArchFragment** — a partial architecture model — using a different analysis strategy:

| Subgraph | Strategy | Focus |
|---|---|---|
| **RAA-A** | SAAM-First (Conservative) | Quality attribute scenarios drive entity selection. Maps requirements to SAAM evaluation scenarios first, then derives entities |
| **RAA-B** | Pattern-Driven | Maps requirements to architectural patterns (layered, event-driven, CQRS, etc.) from a quality-architecture matrix, then derives entities that implement those patterns |
| **RAA-C** | Entity/Relationship-Driven (Bottom-Up) | Extracts C4 entities and relationships directly from requirement text, building the model inductively |

**Hard rules for all subgraphs**:
- Every container must have a valid `parent_system_id` (system must exist in the fragment or running model)
- Every component must have a valid `parent_container_id` (container must exist in the fragment or running model)
- Every relationship must carry an explicit `diagram_scope` consistent with endpoint types:
  - System/Person/ExternalSystem endpoints → `context`
  - Container endpoints → `container`
  - Component endpoints → `component`

#### 6c. Judge (Per-Batch Reconciliation)

The judge receives all subgraph fragments and performs:

1. **SAAM Scenario Scoring**: Score each fragment using ARLO quality weights. Select the highest-scoring fragment as the primary
2. **Entity Deduplication**: Per entity type. Canonical ID normalisation (lowercase, snake_case). When two fragments propose the same entity, merge descriptions (longest wins) and retain technology annotations from any fragment that provides one. Detect hierarchy conflicts (same ID, different parent)
3. **Relationship Deduplication**: Canonical key = `(source_id, target_id, interaction_type)`. Prefer higher-SAAM fragment on conflict. Validate scope consistency across fragments
4. **Coverage Union with Orphan Prevention**: Any entity from a non-selected fragment with SAAM coverage > 0 is considered for addition. Before adding, verify its parent exists in the merged output. Orphans are recorded as `coverage_gap` open questions, not added
5. **Residual Scan**: Before discarding losing fragments, check for entities with unique SAAM coverage not present in the winner
6. **Tree Assembly**: Nest containers into systems, components into containers, distribute relationships by scope. This step is deterministic (no LLM)
7. **Model Update**: Merge the assembled result into `running_arch_model`, advance `batch_cursor`

**Reduced-confidence handling**: For incoherent batches, apply 0.5× weight multiplier to all SAAM scores

### Phase 7 — Human Review Gate

**Responsibility**: Surface accumulated open questions to a human reviewer before final merge. This phase is **optional** — it can be configured to pass through immediately, leaving the judge to resolve everything autonomously.

#### 7a. Open Question Classification

Before any interrupt is issued, all open questions collected across Phase 6 are classified by resolution owner:

| Type | Resolution Owner | Rationale |
|---|---|---|
| `change_risk` | `human_preferred` | Involves organisational or process tradeoffs that require domain knowledge beyond the requirements text |
| `high_coupling` | `human_preferred` | Architectural tradeoff with business impact — e.g., whether tight coupling is acceptable given delivery constraints |
| `coverage_gap` | `human_preferred` | Whether a missing requirement is an intentional scoping decision or an oversight — only a stakeholder can confirm |
| `contention` | `judge_resolvable` | Two fragments propose competing entities; traceable SAAM scores provide a determinate tiebreaker |
| `tie` | `judge_resolvable` | Equal-scoring fragments; resolvable by secondary heuristics (longest description, more requirement links) |
| `hierarchy_conflict` | `judge_resolvable` | Parent–child inconsistency; structurally resolvable by preferring the higher-SAAM fragment's placement |
| `scope_conflict` | `judge_resolvable` | Diagram scope inconsistency; deterministically correctable from endpoint entity types |

#### 7b. Review Payload Emission

The `human_review_payload` channel is populated with:
- All open questions (both owner classes), each with their `id`, `type`, `resolution_owner`, `description`, and `context`
- A read-only summary of the current merged model (entity counts, system names, top open questions by impact)
- Suggested resolutions pre-computed by the judge for `judge_resolvable` questions (so the human can approve rather than compose)

#### 7c. Interrupt Decision

- **`review_mode = interactive`**: The pipeline issues a **LangGraph interrupt** at this node. Execution is suspended and the `human_review_payload` is returned to the caller. The caller is expected to resume the graph with an updated `human_answers` dict (question ID → resolution text). A `review_timeout_seconds` parameter (default: 300 s) is injected at runtime; if the caller does not resume within this window, the pipeline auto-resumes with `human_answers = {}`.
- **`review_mode = autonomous`**: No interrupt is issued. `human_answers` remains an empty dict and Phase 8 proceeds immediately.

> [!IMPORTANT]
> Human answers are **authoritative for any question they address**. If a human provides an answer for a `judge_resolvable` question, that answer overrides the judge's pre-computed suggestion. The judge must never silently ignore a provided answer.

#### 7d. What the Human Must Not Be Asked to Do

The review payload must **not** require the human to:
- Write C4 JSON or entity definitions — answers are free-text guidance (e.g., "treat CRDT as a first-class component inside the Sync Container")
- Resolve `judge_resolvable` types unless they choose to — these are pre-answered and shown for approval only
- Address every question — unanswered questions are resolved by the judge in Phase 8

---

### Phase 8 — Final Merge

**Responsibility**: Combine all batch results into a single validated architecture model, resolve all remaining open questions, and process any requirements that were never included in any batch.

1. Apply the same deterministic merge algorithm globally across all batch fragments
2. **Open Question Resolution** — for each open question:
   - If a human answer exists in `human_answers`: apply it as the authoritative resolution. Set `assumption_flag = false`
   - If no human answer exists and `resolution_owner = judge_resolvable`: apply the judge's pre-computed suggestion. Set `assumption_flag = false`
   - If no human answer exists and `resolution_owner = human_preferred`: the judge makes a **documented assumption** based on the requirement text and SAAM context. Set `assumption_flag = true`. The assumption text is written into the question's `resolution` field and emitted in the final output for traceability
   - The judge is **not permitted to silently skip** any open question — every question must have a `resolution` value before Phase 8 completes
3. **Residual Requirements Pass** — after the batch fragments are merged and the running model is finalised, the judge receives the `unprocessed_requirements` list and the complete merged architecture model. For each unprocessed requirement the judge must:
   - Determine whether the requirement can be **meaningfully and accurately represented** in a C4 diagram (i.e., it implies an architectural element or relationship that belongs in the context, container, or component view)
   - **If representable**: decide which existing entity the requirement enriches (adding it to the entity's `requirement_ids` and updating its description if necessary), or propose a new minimal entity (system, container, component, person, or external system) and its relationship(s) — subject to the same hierarchy and scope rules as any other entity
   - **If not representable**: the requirement is logged as a `coverage_gap` open question with a brief rationale (e.g., process/tooling concern, out-of-scope business rule, or ambiguous phrasing with insufficient architectural signal). It is **not** added to the model
   - The judge must decide **per requirement** — bulk acceptance or bulk rejection is not permitted
   - Each accepted entity or enrichment must pass the same deduplication check applied to batch-produced entities (Section 6)
4. Generate the diagram manifest by traversing the hierarchy (now including any entities added in the residual pass)
5. Validate the output against the C4 JSON schema
6. Write to the orchestrator-provided output path

> [!NOTE]
> The residual pass is a **single-LLM judge call** (not a parallel fan-out) since the merged model already provides the constraint context. The judge should be given the full `unprocessed_requirements` list, the complete merged architecture model, and the same reference excerpts (C4, entity extraction guidelines) as the subgraphs. There is no requirement for all residual requirements to be accepted — the judge's role is discernment, not forced inclusion.

## 6. Deduplication Pass (SAAM Audit Recommendation)

> [!IMPORTANT]
> The SAAM audit identified that merging three independent LLM generation runs (`raa_a`, `raa_b`, `raa_c`) **without a cross-fragment deduplication pass** produces redundant containers. The audit found 7 duplicate container groups in a 50-requirement test, inflating the container count from ~25 to 35.

### 6.1 Required Deduplication Behavior

The judge's merge algorithm (Phase 6c, step 2) must enforce:

- **Semantic deduplication**: Not just exact ID matches, but entities with similar names and overlapping `requirement_ids` should be flagged as potential duplicates
- **Cross-batch deduplication**: The final merge (Phase 7) must perform a global deduplication pass across all batch outputs, not just within a single batch
- **Confidence recalibration**: Duplicated entities should not receive uniform `saam_score: 1.0`. Score should reflect the deduplication risk and the entity's component-level completeness

### 6.2 SAAM Score Calibration

Reserve `saam_score: 1.0` only for entities that have:
1. A component diagram (component-level decomposition)
2. No functional overlap with another entity
3. All direct SAAM scenarios passing

## 8. Embedding Strategy

### 8.1 RAA Owns All Embeddings

The older ARLO version generates embeddings internally for its K-Means clustering step but does **not** persist them to any external store or include them in `ARLOOutput`. Consequently, RAA must generate **all** embeddings from scratch — both ASR (from `condition_text`) and non-ASR (from requirement description text).

This is actually a cleaner ownership model: RAA is the sole writer and reader of the embedding databases, eliminating cross-agent write coordination concerns.

### 8.2 Why Embeddings Are Outside State

Embedding vectors (1024 floats per requirement) would inflate every checkpoint snapshot and provide no benefit — they are immutable once computed. SQLite persistence decouples embedding storage from graph state.

### 8.3 Model Requirement

Both ASR and non-ASR embeddings **must** use the same embedding model and version (`mixedbread-ai/mxbai-embed-large-v1`, 1024-dimensional, CPU-only via FastEmbed) to preserve the shared vector space for meaningful cosine similarity comparisons. This is the same model ARLO uses internally — the vectors will be semantically compatible even though RAA generates them independently.

### 8.4 Access Pattern

Nodes that need embeddings query SQLite directly by requirement ID. They load per-batch or per-group, not the full corpus — keeping memory proportional to batch size, not total requirement count.

### 8.5 Redundant Computation Note

ARLO already computes ASR embeddings internally, so RAA's ASR embedding step re-does this work. This is an acceptable tradeoff: the alternative — modifying ARLO to persist embeddings — would require changes to the upstream agent. The embedding computation is O(n) and runs on CPU, so the cost is modest.

## 9. Checkpointing and Crash Recovery

### 9.1 Checkpoint Mechanism

SQLite-based checkpointing persists the full graph state after every execution step. This enables resumption from the last committed checkpoint without re-executing completed batches.

### 9.2 Resume Logic

- `batch_cursor` is the authoritative resume marker
- If a checkpoint exists with `batch_cursor > 0`, resume from that batch
- If `embeddings_ready = true` in checkpoint, skip the preparation phase
- Completed subgraph outputs within an interrupted batch are preserved as pending writes — only failed subgraphs re-execute

### 9.3 Thread Identity

Each execution is identified by a deterministic thread ID derived from the ARLO output hash, ensuring the same run can be targeted for resume or monitoring.

## 10. Failure Modes

| Risk | Mitigation |
|---|---|
| Condition group has semantic drift | Coherence gate splits or flags with reduced confidence |
| No good non-ASR matches for a group | Allow empty non-ASR list; batch proceeds with ASRs only |
| Cross-batch entity naming collisions | Running model constraint injection + entity deduplication in judge |
| Overlap exceeds 3 bridge requirements | Hard cap enforced |
| Judge discards a useful artifact | Residual scan before fragment discard |
| Incoherent batch degrades model | Reduced-confidence flag + 0.5× SAAM weight multiplier |
| Reconciliation LLM introduces new conflicts | Reconciliation pass scoped strictly to listed open questions; output validated against schema |
| ASR embedding DB missing | Preparation node rebuilds from scratch (RAA is the sole owner) |
| Non-ASR embedding DB missing | Preparation node rebuilds from scratch |
| Embedding text hash mismatch | Recompute only stale embeddings |
| Process killed mid-pipeline | SQLite checkpoint — resume from last committed step |
| Duplicate entities across subgraph runs | Cross-fragment deduplication in judge merge (Section 6) |
| Unprocessed requirements contain critical architecture signal | Residual pass in Phase 8 ensures the judge reviews every unprocessed requirement against the merged model before finalisation |
| Judge bulk-accepts unrepresentable requirements | Per-requirement decision mandate enforced in Phase 8 — bulk accept/reject is explicitly prohibited |
| Residual pass introduces hierarchy violations | Accepted residual entities pass the same parent-resolution and deduplication checks as batch-produced entities |
| Human reviewer never responds (timeout) | Pipeline auto-resumes after `review_timeout_seconds`; judge resolves all unanswered questions autonomously with `assumption_flag = true` |
| Human answer contradicts the merged model structure | Human answers are free-text guidance — the judge interprets and applies them within C4 structural constraints; structurally impossible instructions are logged as a new `scope_conflict` open question and resolved by the judge |
| `interactive` mode used in automated CI/CD pipelines | Callers must set `review_mode = autonomous` for non-interactive environments; the pipeline does not auto-detect this |
| Human answer addresses a question that no longer exists (stale ID) | The judge ignores orphaned answers and logs a warning; this does not block execution |

## 11. Architectural Recommendations (from SAAM Audit)

### 11.1 Critical — Must Address

| # | Issue | Recommendation |
|---|---|---|
| 1 | **Container duplication** — merging 3 independent runs without semantic dedup produces redundant containers | Add semantic deduplication (not just ID matching) in the judge merge. Apply a global dedup pass in final merge |
| 2 | **Uniform SAAM scores** — `saam_score: 1.0` on all entities suppresses design debt signal | Calibrate scores based on component coverage, functional overlap, and scenario pass rate |
| 3 | **CRDT/consistency modelled as labels** — CRDT appears only as a relationship annotation, not a component | Subgraph prompts should promote significant cross-cutting concerns from relationship labels to first-class components |
| 4 | **Compliance assertions without enforcement** — data residency stated in descriptions but not modelled architecturally | Entity extraction guidelines should require enforceable components (Policy Enforcer, Compliance Gate) for regulatory claims |

### 11.2 Important — Should Address

| # | Issue | Recommendation |
|---|---|---|
| 5 | **50% requirements coverage gap** — 25/50 test requirements had no traceable component | This is a scope issue, not an architecture error. The functional gaps represent missing feature domains. Recommendation: extend the input requirements or explicitly scope the model's coverage target |
| 6 | **Process requirements in architecture backlog** — e.g., cyclomatic complexity ≤ 15 | Filter out process/tooling requirements during normalisation (Phase 1). They belong in CI/CD, not C4 diagrams |

## 12. Validation Criteria

### 12.1 Structural Integrity

- Every container is nested inside a system; every component inside a container
- No entity ID appears at more than one hierarchy level
- Every relationship's `source_id` and `target_id` resolve to an entity in the model
- Every relationship's `diagram_scope` matches its endpoint types

### 12.2 Manifest Completeness

- Manifest length = `(2 × systems) + (total containers)`
- One context entry and one container entry per system
- One component entry per container

### 12.3 Orphan Prevention

- A component with an unresolvable `parent_container_id` must not appear in the output — it is recorded as a `coverage_gap` open question instead
- Same rule for containers with unresolvable `parent_system_id`

### 12.4 Cross-Batch Consistency

- Same canonical ID across batches must not have contradictory relationship directions
- Entity descriptions must not conflict (longest-description-wins rule)

### 12.5 Residual Requirements Accounting

- Every requirement in the original input set must appear in **exactly one** of the following: a batch's `requirement_ids`, the `unprocessed_requirements` channel, or as the source of an accepted residual entity/enrichment
- No requirement may silently disappear — all unprocessed requirements are accounted for either by acceptance into the model or by a `coverage_gap` open question entry
- The open question entry for each rejected residual requirement must include: the requirement ID, its description text, and a one-sentence rationale for exclusion

### 12.6 Open Question Resolution Completeness

- Every `OpenQuestion` in the output must have a non-null `resolution` value
- Every question resolved without human input and with `resolution_owner = human_preferred` must have `assumption_flag = true`
- The output's `open_questions` list must distinguish resolved questions (with resolution text) from unresolvable ones — there must be **zero** questions with a null `resolution` after Phase 8 completes
- If `review_mode = interactive` and `human_answers` is non-empty, the number of questions with `assumption_flag = false` and `resolution_owner = human_preferred` must equal the number of matching IDs in `human_answers`
