# C4 Model — Adapted for RAA

## Purpose

Defines the C4 model levels, element types, notation rules, and relationship syntax as they apply to the RAA pipeline. This is an authoritative domain reference — no Input/Output sections are needed.

> **Source:** [https://c4model.com](https://c4model.com)
>
> **Adaptation notice:** Adapted for automated pipeline use. Original C4 is a diagramming methodology; here it is used as a structural schema for entity extraction and relationship mapping by LLM-based subgraphs.

## Three C4 Levels

The RAA pipeline operates at three C4 levels. Each level restricts which element types may appear.

### Context
The top-level system view showing software systems, persons (human actors), and external systems. Shows system-to-system and system-to-actor interactions. Elements at this level: **system**, **person**, **external_system**.

### Container
The deployable-unit view within a single system. Shows web applications, databases, message brokers, services, and other runtime-deployable units. Elements at this level: **container**, and the actors (person, external_system) those containers interact with.

### Component
The internal building-block view within a single container. Shows modules, workers, controllers, libraries, and other code-level decompositions. Elements at this level: **component**, and adjacent containers or external systems those components interact with.

## Element Type Definitions

### System
A software system under design. Represented by `ArchSystem` in the state schema. A system is the root of a C4 hierarchy branch — it contains containers, which contain components. Every system must have a canonical ID (lowercase, snake_case), a human-readable label, and a short description of its purpose.

### Container
A deployable unit within a system. Represented by `ArchContainer`. Must carry a `parent_system_id` linking it to its owning system. Must specify technology stack annotation when determinable from requirements.

### Component
An internal building block within a container. Represented by `ArchComponent`. Must carry a `parent_container_id` linking it to its owning container. Must specify technology stack annotation when determinable.

### Person
A human actor. Represented by `ArchPerson`. A flat leaf entity; never nested. Referenced by ID from relationships.

### External System
A software system outside the design scope. Represented by `ArchExternalSystem`. A flat leaf entity; never nested. Must carry technology annotation when known. Referenced by ID from relationships.

## Notation Rules

1. **Labels:** Every element must have a short, human-readable label. Labels need not be unique across the model but should be unambiguous in context.
2. **Descriptions:** Every element must have a description explaining its purpose in the architecture. One to three sentences.
3. **Technology annotations:** Containers and components must specify technology stack when determinable. Leave as `null` when genuinely ambiguous — never guess. External systems should carry technology when known.
4. **Relationship direction:** Every relationship must state direction (source to target) with a short interaction description as a verb phrase (e.g. "sends payment events to").
5. **Diagram scope:** Every relationship carries an explicit `diagram_scope` value: `context`, `container`, or `component`. Scope is determined by endpoint types (see C4_Level_Mapping.md).
6. **Boundary grouping:** Entities at the same level belonging to the same parent form a natural boundary. Relationships crossing boundaries must justify the crossing in their interaction description.
7. **Level-mixing prohibition:** A context diagram must not show components. A container diagram must not show components from unrelated containers. Each diagram level restricts which element types are visible.

## Relationship Syntax

Every directed relationship (`ArchRelationship`) has these fields:
- `source_id` — canonical ID of the source element.
- `target_id` — canonical ID of the target element.
- `interaction_type` — short verb phrase.
- `technology` — protocol or technology, or `null`.
- `requirement_ids` — list of requirement IDs that imply this relationship.
- `source_fragment` — which subgraph produced it.
- `diagram_scope` — one of `context`, `container`, `component`, assigned by endpoint types.
